# v1.21 / Plan 说明行可读性

## 版本号

v1.21

## 版本分配依据

- 当前体验优化阶段已完成到 v1.20。
- v1.20 补齐了 Root iPad sidebar 的可访问性语义。
- Plan 页面中 `Current Build` 与 `Platform Status` 仍各有一条裸 `Label` 说明，缺少统一的最小行高、换行约束和显式 VoiceOver label/value/hint。

## 背景

Plan 页面已经有 `PlanStatusSummaryRow`、`PlanStatusRow`、`PlanStatusBadge` 等可读状态组件，用于表达 Local plan、StoreKit 未配置、平台状态、Mac readiness、能力矩阵、权益规则和可用性说明。

当前 `Current Build` 和 `Platform Status` 的尾部说明仍是普通 `Label`：

```swift
Label("Purchases, restore, receipts, subscriptions, and entitlements are not enabled in this build.", systemImage: "exclamationmark.triangle")
    .foregroundStyle(SciFiTheme.secondaryText)
```

```swift
Label("Mac support requires Xcode platform changes, a native backend Mac/Catalyst slice, signing decisions, and dedicated UI validation.", systemImage: "checklist")
    .foregroundStyle(SciFiTheme.secondaryText)
```

这些说明承载付费和 Mac 可用性边界，是 Plan 页面最容易被误解的文本之一。本轮只把它们提取为统一的说明行组件，提升 iPad/compact/Dynamic Type 下的阅读和 VoiceOver 语义，不改变任何付费、平台或业务逻辑。

## 目标

- 新增私有 `PlanNoteRow` 或等价小组件。
- 用该组件替换 `Current Build` 与 `Platform Status` 中的两条裸 `Label` 说明。
- 保留原有说明文案事实：
  - 当前 build 未启用 purchases、restore、receipts、subscriptions、entitlements。
  - Mac support 仍需要 Xcode 平台、native backend Mac/Catalyst slice、签名决策和专门 UI 验证。
- 说明行应：
  - 使用真实 `Label` 和 SF Symbol，不改成纯自绘图标。
  - 使用 `.font(.callout)` 或当前 Plan 说明文本一致的系统字体。
  - 文本可纵向换行，避免 Dynamic Type 下横向压缩。
  - 有 44pt 最小高度。
  - 提供明确 VoiceOver label、value 和 hint。
- 更新 README、`md/flow/flow.md`、`md/flow/flowchart.md`、`update_log.md`，记录 Plan 说明行语义改善。

## 非目标

- 不新增购买、恢复、收据、订阅、entitlement、product ID、StoreKit 配置或 App Store Connect 行为。
- 不持久化任何 entitlement 或购买状态。
- 不改变当前 Local plan 能力、能力矩阵、权益规则、可用性说明或 Mac readiness 事实。
- 不启用 Mac Catalyst，不修改 `SUPPORTED_PLATFORMS`、`SUPPORTS_MACCATALYST`、Xcode project、签名或分发配置。
- 不修改 native XCFramework、native bridge、SwiftData schema、文件存储、生成流程或下载流程。
- 不改 Root 导航、compact `TabView`、iPad `NavigationSplitView`、Gallery layout 或其他页面行为。
- 不修改 CI workflow。

## 当前架构依据

- `RootContentView.swift` 内的 `PlanView` 属于 UI 展示层，只能表达状态、限制和用户反馈。
- Plan 当前是付费能力和平台能力的信息架构基线，不是 StoreKit 接入点。
- 现有 `PlanStatusSummaryRow` 和 `PlanStatusRow` 已经用 label/value/status badge 明确表达状态；说明文本应保持同一套可读性和可访问性质量。
- SwiftUI 可访问性要求：重要说明不能只依赖颜色或隐含 row 文本，应给 VoiceOver 明确 label/value/hint；交互或重要行高应满足 44pt 阅读/触控基线。
