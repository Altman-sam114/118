# v1.8（Plan 状态行可读性）

## 版本号

v1.8

## 版本分配依据

v1.6 把 Plan 改为 iPad regular 双栏阅读布局，v1.7 增加 Entitlement Rules，使 Plan 的状态行数量继续增加。当前 Mac Readiness、Capability Matrix、Entitlement Rules 三组 row 都使用 `LabeledContent` 尾部状态标签，长状态在 iPhone 和 Dynamic Type 下容易被压缩、缩小或挤到尾部。本轮继续推进人工总目标中的 iPad/UI/design 优化：把 Plan 状态行改成更稳定、可换行、文本+图标+边框明确表达的状态徽章布局。

## 背景

Plan 当前是信息密度最高的规划页面：同时承担本地计划、付费候选、权益规则和 Mac readiness 的事实说明。它必须保持 truthful，但也要在 iPhone compact、iPad regular 和较大 Dynamic Type 下保持可读。状态不能只靠颜色表达，也不应该为了放进尾部列而缩小过多。由于本轮只优化 UI 展示结构，不需要外部 StoreKit、Mac Catalyst 或 native backend 依赖。

## 目标

- 在 `RootContentView.swift` 中为 Plan 增加共享状态展示结构，减少 `MacReadinessRow`、`CapabilityRow`、`EntitlementRuleRow` 的重复。
- 用一个共享的 `PlanStatusRow` 和 `PlanStatusBadge` 展示：
  - 左侧/上方：行图标、标题和详情。
  - 下方或同一垂直流内：状态徽章，包含状态文字、SF Symbol 图标、颜色和描边。
- 状态徽章必须使用文字 + 图标 + 颜色，不只靠颜色表达状态。
- 去掉这些状态行里为尾部状态挤压而使用的 `lineLimit(2)` 和 `minimumScaleFactor(0.85)`，让 Dynamic Type 可以自然换行。
- Current Build 与 Platform Status 也应使用一致的状态徽章，不再使用尾部 `LabeledContent` 状态文本。
- iPad regular 在 accessibility Dynamic Type 下应回退为单列堆叠；普通双栏应给列设置真实最小宽度，避免 `ViewThatFits` 误判“可放下”但列过窄。
- Plan overview 在 accessibility Dynamic Type 下应把固定图标与文案改成纵向排列，避免横向挤压正文。
- 保持 compact `Form` 和 iPad regular 双栏布局语义不变。
- 不改变任何 plan 状态事实、不新增 StoreKit 行为、不启用 Mac Catalyst。
- 同步 README、flow、flowchart、update_log。

## 非目标

- 不接入 StoreKit，不新增 product ID、价格、SKU、购买、恢复、receipt、subscription 或 entitlement 持久化。
- 不启用 Mac Catalyst，不修改 `SUPPORTED_PLATFORMS`、`SUPPORTS_MACCATALYST`、native XCFramework 或 Xcode project。
- 不修改 Generate、Models、Gallery、Prompts、SwiftData schema、文件存储、native bridge ABI 或 CI workflow。
- 不做大范围视图拆文件或 Xcode project 文件引用维护。
- 不做 simulator smoke 或截图验收，除非人工另外要求或本地验证暴露必须复现的问题。

## 当前架构依据

- `PlanView` 当前在 `LocalDiffusion/Views/RootContentView.swift` 内，是纯 SwiftUI 展示层。
- Plan 的三组状态 row 数据分别是 `MacReadinessItem`、`PlanCapabilityItem` 和 `PlanEntitlementRuleItem`，它们都有 `title`、`detail`、`status` 和 `systemImage`。
- 对应 status enum 都已经提供 title、systemImage 和 color，可以通过一个共享 token 或 computed property 传给统一 row。
- SwiftUI 设计要求：避免固定屏幕宽度，不使用 `UIScreen.main.bounds`；状态不能只靠颜色；支持 Dynamic Type；图标装饰应隐藏或搭配文本。
- 带 footer 的 `Section` 应继续使用显式 `Section { } header: { } footer: { }` 形式。

## 实现步骤

1. 阅读必读文件、本提示词、`RootContentView.swift` 和相关文档。
2. 同步最新 `origin/main`，确认当前分支是 `main` 且无无关改动：

```bash
git fetch origin
git status --short --branch
git rev-parse HEAD origin/main
```

3. 在 `RootContentView.swift` 中新增小型共享类型，建议为：
   - `PlanStatusToken`
   - `PlanStatusBadge`
   - `PlanStatusRow`
4. 为 `PlanCapabilityStatus`、`PlanEntitlementRuleStatus`、`MacReadinessStatus` 增加 `token` computed property，或用等价方式把 `title`、`systemImage`、`color` 传给共享 row。
5. 将 `MacReadinessRow`、`CapabilityRow`、`EntitlementRuleRow` 改为薄 wrapper，内部调用共享 `PlanStatusRow`。
6. 为 Current Build 与 Platform Status 增加等价 summary row，复用 `PlanStatusBadge`。
7. `PlanStatusRow` 应使用垂直布局，避免尾部状态挤压正文：
   - `Label` 展示标题和行图标。
   - `Text` 展示详情，允许换行。
   - `PlanStatusBadge` 展示状态。
   - 设置 `.frame(maxWidth: .infinity, alignment: .leading)`。
   - 可使用 `.accessibilityElement(children: .combine)` 让 VoiceOver 一次读完整行。
8. `PlanStatusBadge` 应使用 `Label(status.title, systemImage: status.systemImage)`，并加轻量背景和描边，保持暗色科幻风格。
9. 在 `PlanView` 中读取 `dynamicTypeSize`：
   - accessibility size class 下 regular 布局直接使用单列。
   - 普通 regular 双栏给左右列设置合理 `minWidth`。
   - Plan overview 在 accessibility size class 下把图标和文字改为纵向排列。
10. 同步文档：
   - `README.md`：说明 Plan 状态矩阵/规则使用可读状态徽章。
   - `md/flow/flow.md`：更新 Plan UI 职责，注明状态徽章不只靠颜色表达。
   - `md/flow/flowchart.md`：更新 Plan 节点。
   - `update_log.md`：追加 v1.8 记录。

## 验收标准

- Current Build、Platform Status、Mac Readiness、Capability Matrix、Entitlement Rules 的状态展示使用共享状态徽章。
- 状态仍由文字、图标和颜色共同表达，不只靠颜色。
- 状态文字不再依赖 `minimumScaleFactor` 才能塞进尾部列；较大 Dynamic Type 下可以自然换行。
- iPad regular 在 accessibility Dynamic Type 下回退单列，普通双栏有真实最小列宽。
- Plan overview 在 accessibility Dynamic Type 下不再让固定图标横向挤压文案。
- compact 和 regular Plan 内容顺序、事实语义和警示文案保持不变。
- 不新增 StoreKit、Mac Catalyst、Xcode project、native backend、SwiftData 或文件存储改动。
- 本地轻量检查和 Swift parse 通过；建议的 iPhoneOS build 通过或明确记录沙箱限制。
- `origin/main` 最新 v1.8 run 成功，Agent C 下载并核对结果包通过。
