# v0.6（CI 资产完整性校验）

## 版本号

v0.6

## 版本分配依据

v0.4 已建立 `main` 直推和 GitHub Actions 结果包流程。v0.5 正在把被 `.gitignore` 排除的 `LocalDiffusionNative.xcframework` 改为从 GitHub Release `native-backend-current` 恢复，但当前工作树显示 v0.5 相关修改仍未提交到 `origin/main`。本轮目标不是扩展产品功能，而是继续加固 v0 治理和 CI native 资产链路，因此分配为 v0.6。

前置要求：Agent B 启动本轮前必须先确认 v0.5 是否已完成提交、push 和 Agent C 结果包验收。若工作区仍有 v0.5 未提交修改，不得把 v0.6 与 v0.5 混在同一个提交里；必须先按人工指令完成或退回 v0.5，再执行本提示词。

## 背景

当前项目进展：

- v0.1 已建立 SwiftUI + SwiftData + 文件存储 + native backend bridge 的本地图片生成主链路。
- v0.2 至 v0.4 已建立多 Agent 文档体系、Agent C 验收规则、`main` 直推和 GitHub Actions 云端结果包流程。
- v0.5 的目标是让 CI 在构建前从 Release asset 恢复 `LocalDiffusionNative.xcframework.zip`。
- 当前仍存在关键风险：CI 能下载 zip 不等于下载到预期资产；Release asset 可能缺失、过旧、被替换，或者与当前 bridge ABI 不匹配。

Agent A 本轮本地观察到 `/private/tmp/LocalDiffusionNative.xcframework.zip` 存在，大小约 67 MB，SHA-256 为 `a1446f367c518155fac0ac268f8ee30b882961776fbbe721588d64db00b1e8d8`。这只是当前机器线索，Agent B 不得直接把它当作云端事实；必须对实际待上传或已下载的 Release asset 重新计算并核对。

## 目标

- 为 native Release asset 建立一个小型、可 git 跟踪的元数据文件，记录 release tag、asset 名称、SHA-256、生成/刷新时间、生成来源和人工维护说明。
- 修改 `.github/workflows/ci-results.yml`，在下载 `LocalDiffusionNative.xcframework.zip` 后先做 SHA-256 校验，校验失败时停止后续 native preflight 和 build，并把失败原因写入结果包。
- 将 asset 校验日志、期望摘要、实际摘要和元数据路径纳入 `ci-artifact-manifest.json` 与 `projectSpecificReports`。
- 同步 `md/test/test.md`、`md/flow/flow.md`、`md/flow/flowchart.md`、README 和 `update_log.md`，明确 Release asset 刷新后的校验和验收规则。
- 保持 native binary 不进入 git，保持 UI、SwiftData、文件存储和推理协议边界不变。

## 非目标

- 不把 `LocalDiffusionNative.xcframework` 或 zip 资产提交进普通 git。
- 不修改 Swift 业务源码、SwiftUI 页面、SwiftData schema、Xcode project 或 native bridge ABI。
- 不重建 stable-diffusion.cpp 包装，除非发现当前 Release asset 与本轮校验无法对应。
- 不设计 PR、`develop`、`smalldata_test` 或其他分支流程。
- 不把本轮作为真实 GGUF 端到端生成验收；v1.0 再处理真机或真实模型运行闭环。

## 当前架构依据

- `ImageGenerationBackend` 仍是 UI 到推理的唯一边界。
- `LocalDiffusionNative.xcframework` 是生成物，受 `.gitignore` 排除，不适合直接进入源码仓库。
- `.github/workflows/ci-results.yml` 是 `main` push 后的云端验证入口。
- `Scripts/check-native-backend.sh` 负责检查恢复后的 XCFramework 目录、头文件、链接配置、导出符号和模拟器架构。
- 资产完整性校验应发生在 native preflight 和 `xcodebuild` 之前；校验失败应被判定为云端依赖或资产维护问题，而不是 Swift 业务代码失败。

## 实现步骤

1. 按 `AGENTS.md` 顺序读取必读文件、v0.5 提示词、当前 workflow、native backend 脚本和本提示词。
2. 同步并检查仓库状态：
   - `git fetch origin`
   - `git switch main`
   - `git pull --ff-only origin main`
   - `git status --short`
   - 若存在 v0.5 或其他无关未提交修改，停止并记录阻塞，不得混合提交。
3. 新增一个被 git 跟踪的 native asset 元数据文件，建议路径为 `NativeBackend/StableDiffusionCpp/native-backend-asset.json`。字段至少包含：
   - `releaseTag`
   - `assetName`
   - `sha256`
   - `generatedAt`
   - `sourceDescription`
   - `refreshRequiredWhen`
4. 对实际要上传或已上传的 `LocalDiffusionNative.xcframework.zip` 重新计算 SHA-256，并写入元数据。若无法访问或无法确认 Release asset，记录阻塞，不得虚构摘要。
5. 修改 `.github/workflows/ci-results.yml`：
   - 设置元数据路径环境变量。
   - 下载 asset 后读取元数据中的 `sha256`。
   - 用 `shasum -a 256` 或等价命令计算下载 zip 的实际摘要。
   - 将期望值、实际值和比较结果写入 `ci-results/native-backend-asset.log`。
   - 摘要不一致时退出失败，并确保 artifact 仍上传。
   - manifest 中加入 asset metadata path、expected sha、actual sha 和 `nativeBackendAssetOutcome`。
6. 更新测试规范和流程文档：
   - `md/test/test.md` 说明本地如何校验 JSON 和 workflow，Agent C 如何核对 asset log。
   - `md/flow/flow.md` 和 `md/flow/flowchart.md` 加入“下载后校验 SHA-256，再解压和 preflight”的 CI 资产流程。
   - README 简短说明 native Release asset 有校验元数据。
   - `update_log.md` 记录 v0.6 的目标、关键文件、验证要求和遗留风险。
7. 本地运行轻量检查。
8. 创建版本提交并推送 `origin/main`。
9. 等待 GitHub Actions，记录 run id、run attempt 和 artifact 名称，交给 Agent C 下载结果包核对。

本地轻量检查至少运行：

```bash
git diff --check
plutil -lint LocalDiffusion.xcodeproj/project.pbxproj
ruby -e 'require "yaml"; YAML.load_file(".github/workflows/ci-results.yml"); puts "yaml ok"'
python3 -m json.tool NativeBackend/StableDiffusionCpp/native-backend-asset.json >/dev/null
```

若本轮只改 workflow、JSON 和文档，不需要本机 Swift parse、native preflight、simulator smoke 或完整 `xcodebuild`。这些由云端 run 验证，或在 Agent C 发现云端失败后再按风险补充。

如果 Agent B 实际刷新了 zip asset，本地还必须运行：

```bash
shasum -a 256 /path/to/LocalDiffusionNative.xcframework.zip
```

并在最终输出中说明该摘要是否与 `native-backend-asset.json` 一致。

## 验收标准

- `native-backend-asset.json` 是 git 跟踪文件，字段完整，JSON 可解析。
- workflow 下载 Release asset 后会校验 SHA-256，校验失败时 `native-backend-asset.log` 和 `ci-failure-summary.md` 能定位原因。
- manifest 记录 asset 校验结果、元数据路径、期望摘要和实际摘要。
- 本地轻量检查全部退出 0。
- `git push origin main` 后触发 v0.6 Actions run。
- Agent C 下载 `/private/tmp/localdiffusion-c-review-<run_id>/` 下的最新结果包，确认 manifest 的 branch、commitSha、runId、runAttempt 与 `origin/main` 最新状态一致，并核对 `native-backend-asset.log`。

## 风险和禁止项

- 禁止把本地 `/private/tmp` 观察值直接当作 Release asset 事实。
- 禁止提交 zip、XCFramework 或其他大二进制生成物。
- 禁止在 checksum 不一致时继续声称 native preflight 或 build 代表资产可用。
- 禁止混合 v0.5 未完成修改和 v0.6 提交。
- 禁止用旧 run、旧 artifact 或 Agent B 文字汇报替代 Agent C 结果包核对。
- 若远端凭据、GitHub Release 权限或 Actions 权限不足，必须明确记录阻塞，不得伪造上传、run id 或 artifact。
