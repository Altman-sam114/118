# v0.4（云端验证 main 直推流程）

## 版本号

v0.4

## 版本分配依据

人工未指定版本号。本轮是在既有 v0 项目治理体系上，把本地 Agent C 提交验收升级为 `main` 直推、GitHub Actions 云端重验证和 Agent C 结果包复判，属于治理流程小版本升级，因此使用 `v0.4`。

## 背景

当前项目已有 `AGENTS.md`、`update_log.md`、`md/flow/flow.md`、`md/flow/flowchart.md`、`md/test/test.md` 和 `md/prompt/`。旧流程以本机检查和本地提交为主，Agent C 缺少可下载、可追溯的云端结果包作为验收依据。

## 目标

- 写清 Agent A/B/C 召唤前缀和最终回复身份标识。
- 固定默认流程为本地轻量检查、`main` 版本 commit、`git push origin main`、GitHub Actions 云端重验证。
- 让 Agent C 必须下载并核对未加密 CI 结果包后才能验收通过。
- 新增 `.github/workflows/ci-results.yml`，在 `main` push 和手动触发时运行，并上传 manifest、failure summary、JUnit、主构建日志和项目关键报告。
- 同步 `AGENTS.md`、`md/test/test.md`、`md/flow/flow.md`、`md/flow/flowchart.md`、`md/prompt/README.md`、`README.md` 和 `update_log.md`。

## 非目标

- 不修改 Swift 业务源码。
- 不改 UI、SwiftData、文件存储、native bridge ABI 或推理语义。
- 不引入 `smalldata_test`、`develop`、`codeb/...` 或 PR 合并流。
- 不照搬 AITRANS 的漫画探针、模型发布包、业务指标或项目特例。
- 不把旧 artifact、旧 output 或 checkout 自带报告当成本轮云端结果。

## 架构依据

- UI 层仍只依赖 `ImageGenerationBackend`，不直接调用 C bridge。
- native backend 预检仍由 `Scripts/check-native-backend.sh` 承担。
- Xcode 构建入口仍是 `LocalDiffusion.xcodeproj` / `LocalDiffusion` scheme。
- 文档职责仍按 `AGENTS.md`、`update_log.md`、`md/flow/**`、`md/test/test.md`、`md/prompt/**` 和 README 分层。

## 实现步骤

1. 读取必读文件、git 状态、分支、远端、现有 workflow 和构建脚本。
2. 在动文件前记录迁移判断：项目类型、main 状态、CI 现状、可复用规则和不能照搬的 AITRANS 特例。
3. 更新 `AGENTS.md`，加入角色召唤、身份标识、main 直推、云端结果包验收和禁止项。
4. 更新 `md/test/test.md`，明确默认云端重验证、本地轻量检查、结果包内容和 Agent C 下载缓存位置。
5. 更新 `md/flow/flow.md` 与 `md/flow/flowchart.md`，展示 Agent A/B/C -> main push -> Actions -> artifact -> Agent C 复判 -> 追加修复 commit 闭环。
6. 更新 `md/prompt/README.md`，要求 Agent A 提示词包含 CI、main push、artifact 和退回修复要求。
7. 更新 README 的简短协作与云端验证说明。
8. 更新 `update_log.md`，记录本轮是协作流程制度变更，不伪装成业务质量提升。
9. 新增 `.github/workflows/ci-results.yml`，产出未加密 CI 结果包。
10. 跑本地轻量检查；若仓库有 `origin`，提交并 push 后等待 Actions、下载并核对 artifact。

## 测试要求

本地最低检查：

```bash
git diff --check
plutil -lint LocalDiffusion.xcodeproj/project.pbxproj
ruby -e 'require "yaml"; YAML.load_file(".github/workflows/ci-results.yml"); puts "yaml ok"'
```

若需要确认 workflow JSON/JUnit 生成逻辑，可在云端 run 后由 Agent C 下载结果包核对。除非人工明确要求，本轮不默认本机跑完整 `xcodebuild` 或 simulator smoke。

## 文档更新要求

- `AGENTS.md` 保持精简，只写硬规则和流程边界。
- `flow.md` 与 `flowchart.md` 必须同步。
- `test.md` 必须写清本地轻量检查、云端重验证、结果包内容、下载缓存和未跑测试说明。
- `prompt/README.md` 必须写清云端阶段的 Agent A 提示词要求。
- `update_log.md` 记录为治理流程升级。
- README 只补简短协作入口，不写成第二份 `AGENTS.md`。

## 验收标准

- 文档写清 Agent A/B/C 召唤规则和身份标识。
- 文档写清 `main` 是默认上传、提交、推送和云端验证分支。
- Agent B 默认本地轻量检查、commit、push `origin/main`。
- Agent C 默认下载未加密 CI 结果包并核对 manifest/JUnit/log。
- `.github/workflows/ci-results.yml` 能在 `main` push 和 `workflow_dispatch` 上运行并上传结果包。
- 本地轻量检查通过。
- 若当前仓库没有 `origin` 或无法访问 GitHub Actions，必须明确记录云端试跑阻塞，不能伪造 run 或 artifact。

## 风险和禁止项

- 禁止把本轮当作业务功能质量升级。
- 禁止照搬 AITRANS 项目专属探针和输出。
- 禁止无权限下载 artifact 时声称已核对。
- 禁止把旧报告冒充本轮云端结果包。
- 禁止默认新增 PR、候选分支或多分支发布流程。
