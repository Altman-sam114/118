# v0.3（Agent C 通过后版本提交）

## 版本号

v0.3

## 版本分配依据

人工未指定版本号。本轮是在既有 v0 项目治理文档体系上补充 Agent C 验收后的版本提交规则，属于小版本治理规则更新，因此使用 `v0.3`。

## 背景

当前项目已经建立 `AGENTS.md`、`update_log.md`、`md/prompt`、`md/test`、`md/flow` 的多 Agent 协作体系。原流程停在 Agent C 验收并交给人工复核，缺少“验收通过后自动形成可追踪 git 版本”的硬规则，容易导致完成事项、测试结果和代码状态脱节。

## 目标

- 修改 `AGENTS.md`，明确 Agent C 验收通过后必须按版本号创建 git 提交。
- 明确 Agent C 验收不通过时必须退回 Agent B 修复，不得提交失败版本。
- 规定提交信息格式：`vX.Y: 简要任务名`，正文简要概括完成内容、验证结果和遗留风险。
- 同步 `md/flow/flow.md` 与 `md/flow/flowchart.md` 的 Agent 迭代流程。
- 同步 `update_log.md`、`md/prompt/README.md` 和 README 的维护说明。

## 非目标

- 不修改 Swift 源码。
- 不变更 native backend、SwiftData 模型、UI 或业务逻辑。
- 不为当前文档修改强制创建 git 提交，除非人工明确要求。
- 不改变 `agent.md` 兼容跳转定位。

## 当前架构依据

- 标准入口：`AGENTS.md`
- 历史记录：`update_log.md`
- 当前核心流程：`md/flow/flow.md`
- 流程图：`md/flow/flowchart.md`
- 测试规范：`md/test/test.md`
- Agent A 提示词目录：`md/prompt/README.md`
- 旧引用兼容：`agent.md`

## 实现步骤

1. 阅读 `AGENTS.md`、`update_log.md`、`md/flow/flow.md`、`md/flow/flowchart.md`、`md/test/test.md`、`README.md`。
2. 在 `AGENTS.md` 的基本规则、Agent C 工作流、交付格式和禁止项中加入版本提交规则。
3. 在 `md/flow/flow.md` 加入 Agent 版本迭代流程和验收通过后提交边界。
4. 在 `md/flow/flowchart.md` 的 Agent 迭代图中加入通过/不通过分支、退回 Agent B、git commit 节点。
5. 在 `update_log.md` 追加 v0.3 历史记录。
6. 在 `md/prompt/README.md` 增加 Agent C 通过后提交规则。
7. 更新 README 的 Agent handoff 和维护日志。
8. 运行 `git diff --check`。

## 测试要求

文档-only 修改至少运行：

```bash
git diff --check
```

无需运行 iOS 构建、native backend preflight 或 simulator smoke test；最终输出必须说明未跑业务测试的原因。

## 文档更新要求

- `AGENTS.md` 必须写清楚 Agent C 通过和不通过的分支。
- `flow.md` 与 `flowchart.md` 必须保持一致。
- `update_log.md` 必须记录 v0.3 治理规则变化。
- README 必须保留给后续 Agent 的入口说明。

## 验收标准

- Agent C 通过后提交版本的规则清晰、可执行。
- Agent C 不通过时退回 Agent B 的规则清晰、可执行。
- 提交信息格式包含版本号和简要任务名。
- Mermaid 图能看出通过、退回、提交、人工复核的完整闭环。
- `git diff --check` 通过。

## 风险和禁止项

- 禁止在 Agent C 验收不通过时创建提交。
- 禁止伪造提交哈希或用未提交改动冒充已提交版本。
- 禁止把所有未跟踪文件一概提交，必须只提交本轮相关文件。
- 禁止把历史细节堆进 `AGENTS.md`。
